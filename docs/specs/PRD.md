# 产品需求文档（PRD）

> Product Requirements Document — EasyTransfer
> 版本：0.2.0 | 最后更新：2026-03-25

---

## 1. 产品定位（v0.2 重新定义）

### 1.1 核心理念

> EasyTransfer 不是一个传统的 App，而是一项**卖给 AI Agent 的技能（Skill）**。
>
> 就像人类助理需要学习新技能才能帮雇主搬家一样，
> 用户的 AI 助手（如 OpenClaw、Claude、Open Interpreter 等）需要安装 EasyTransfer 技能，
> 才能帮用户完成电脑迁移。

### 1.2 产品形态

```
传统思维：                          我们的思维：
  开发一个 App → 卖给用户             开发一项技能 → 卖给用户的 AI 助手
  用户学习如何使用 App                用户只需要说一句话
  App 有自己的界面                   AI 助手就是界面
  App 处理所有逻辑                   AI 助手做决策，我们的技能做执行
```

### 1.3 技术形态：MCP Server（模型上下文协议服务器）

我们的技能包以 **MCP Server** 的形式发布。

**什么是 MCP？**
MCP（Model Context Protocol）是 AI Agent 调用外部工具的标准协议。
就像浏览器通过 HTTP 协议访问网站一样，AI Agent 通过 MCP 协议调用工具。
目前 Claude、OpenClaw 等主流 Agent 平台都支持或正在接入 MCP。

**为什么选 MCP？**
- 这是行业正在收敛的标准，不是某一家的私有协议
- 一次开发，所有支持 MCP 的 Agent 都能用
- 用户安装技能的方式标准化（类似安装浏览器插件）
- 未来可能出现"技能商店"，我们可以直接上架

### 1.4 商业模式

```
收入模式：技能调用付费（本质上是卖 token/卖服务）

免费层：
  - 环境扫描（让用户看到效果）
  - 生成迁移计划（让用户知道能迁移什么）
  - 迁移最多 5 个应用 + 1GB 文件

付费层（一次性购买 or 按次付费）：
  - 完整迁移（不限应用数量和文件大小）
  - 预计定价：¥49-99 / 次

我们的成本：
  - AI API 调用费（用于智能规划）≈ ¥2-5 / 次
  - 服务器成本（仅中继，大部分数据走局域网直连）≈ 极低
  - 毛利率：90%+
```

## 2. 问题描述

### 2.1 用户痛点

当用户购买新的 Windows 电脑后，面临一个极其痛苦的过程：将旧电脑上积累的所有软件、配置、文件和凭证迁移到新电脑上。目前这个过程存在以下痛点：

- **耗时巨大**：通常需要 1-3 天才能让新电脑达到旧电脑的工作状态
- **容易遗漏**：总会忘记某些配置或某个小工具，直到用的时候才发现
- **技术门槛高**：很多配置需要专业知识才能正确迁移
- **没有好用的现有工具**：Windows 易传输已被移除，第三方工具要么贵、要么功能不完整

### 2.2 目标用户

**核心用户画像**：已经在使用 AI Agent（如 OpenClaw、Claude Desktop、Open Interpreter）的用户

- 这类用户已经习惯用自然语言让 AI 帮自己完成任务
- 他们愿意为 AI 助手"购买技能"来扩展能力
- 他们对效率有极高要求
- 大概率是开发者或科技爱好者（早期）

**扩展用户**：随着 AI Agent 普及，普通用户也会成为我们的用户

## 3. 用户使用门槛分析与解决方案

### 3.1 门槛一览

```
用户要完成一次迁移，理论上需要：

步骤 1：在旧电脑上有一个 AI Agent        ← 门槛：可能还没装
步骤 2：在新电脑上有一个 AI Agent        ← 门槛：新电脑刚开箱，什么都没有
步骤 3：两台电脑都安装我们的技能包       ← 门槛：安装过程复杂吗？
步骤 4：两台电脑在同一网络              ← 门槛：网络配置
步骤 5：告诉 Agent 开始迁移             ← 没有门槛，说句话就行
```

**最大的门槛是步骤 2：新电脑上什么都没有，怎么装 Agent？**

### 3.2 解决方案：三层递进

#### 方案 A：最轻量 —— "单机扫描 + 云端中转"（推荐 MVP）

```
只需要在旧电脑上安装！新电脑几乎零配置！

流程：
  旧电脑（已有 Agent + 技能）：
    1. 用户说："扫描我的电脑，准备换机"
    2. 技能扫描环境，生成迁移包
    3. 迁移包加密上传到云端（或生成到 U 盘）

  新电脑（全新开箱）：
    4. 用户下载一个单文件 .exe（迁移恢复器，<50MB）
       获取方式：浏览器打开一个短链接，下载运行
    5. 输入迁移码（6位数字）
    6. 自动下载迁移包并执行恢复
    7. 恢复完成后，新电脑上也装好了 Agent + 技能

  用户实际操作：
    旧电脑：说一句话
    新电脑：下载一个 exe + 输入 6 位数字

  → 这就是我们的"一键换机"体验
```

**为什么这个方案好：**
- 新电脑不需要预装任何 Agent
- 用户只需要下载一个小文件
- 6 位迁移码的体验和手机换机一模一样
- 恢复器可以自带精简版 Agent，恢复完成后自动升级为完整版

#### 方案 B：局域网直连（性能最佳）

```
适合：两台电脑都已有 Agent 的高级用户

流程：
  旧电脑 Agent ←── 局域网直连 ──→ 新电脑 Agent

  用户在任意一端说："开始迁移"
  两个 Agent 自动发现对方，完成迁移

  优势：速度最快（千兆局域网 100GB ≈ 15 分钟）
  劣势：需要两端都有 Agent
```

#### 方案 C：远程迁移（跨地域）

```
适合：旧电脑和新电脑不在同一地点

流程：
  旧电脑 → 加密上传到云端 → 新电脑下载恢复

  优势：不受地域限制
  劣势：受上传带宽限制，大文件较慢
  解决：只上传应用列表+配置，文件类让用户用网盘自行迁移
```

### 3.3 推荐的 MVP 策略

```
MVP 只做方案 A（单机扫描 + 迁移码恢复），原因：

1. 用户门槛最低 —— 新电脑不需要任何预配置
2. 开发量最小 —— 不需要实时双端通信
3. 体验最接近"手机换机" —— 扫码/输码即可
4. 能验证核心价值 —— 扫描准不准、恢复好不好用

MVP 之后再加方案 B（局域网直连提速）
```

### 3.4 降低门槛的其他手段

| 门槛 | 解决方案 |
|------|---------|
| 不知道我们的产品 | 和 Agent 平台合作，在技能商店上架 |
| 旧电脑没有 Agent | 提供独立的扫描器（无需 Agent 也能用，但体验略差） |
| 新电脑没有 Agent | 方案 A 的单文件恢复器解决 |
| 安装技能包复杂 | MCP 标准的一行命令安装：`npx easytransfer-mcp` |
| 不信任安全性 | 开源核心代码，端到端加密，安全审计报告 |
| 网络配置复杂 | 自动发现（mDNS），失败时引导手动输入 IP |

## 4. 产品功能

### 4.1 作为 MCP 技能暴露的工具列表

以下是 AI Agent 可以调用的工具（Tools），也是我们技能包的核心 API：

```
工具 1：scan_environment
  描述：扫描当前电脑的完整软件环境
  输入：扫描范围选项（可选）
  输出：结构化的环境画像（JSON）
  耗时：1-5 分钟
  示例调用场景：用户说"看看我电脑上装了什么"

工具 2：analyze_migration
  描述：分析环境画像，生成人类可读的迁移报告
  输入：环境画像 JSON
  输出：迁移项清单（有多少应用、多少文件、哪些能自动迁移）
  耗时：几秒（本地分析）+ 可选 AI 增强分析
  示例调用场景：用户说"哪些东西需要迁移？"

工具 3：create_migration_package
  描述：根据用户确认的迁移计划，打包迁移数据
  输入：迁移计划（含用户选择）
  输出：加密的迁移包（本地文件或云端链接 + 迁移码）
  耗时：取决于数据量
  示例调用场景：用户说"好的，开始打包"

工具 4：restore_from_package
  描述：从迁移包恢复环境到当前电脑
  输入：迁移包路径或迁移码
  输出：恢复进度和结果报告
  耗时：取决于数据量和应用数量
  示例调用场景：在新电脑上运行

工具 5：verify_migration
  描述：验证迁移结果的完整性
  输入：迁移记录
  输出：验证报告（成功/失败/需手动处理的项目）
  示例调用场景：用户说"检查一下迁移是不是都成功了"

工具 6：rollback_migration
  描述：回滚指定的迁移操作
  输入：需要回滚的迁移项 ID
  输出：回滚结果
  示例调用场景：用户说"把刚才安装的 XX 撤销掉"
```

## 5. 非功能需求

### 5.1 性能要求
- 扫描阶段：5 分钟内完成（普通电脑，约 100 个应用 + 100GB 文件）
- 迁移包生成：压缩比 ≥ 40%
- 恢复阶段：应用安装速度取决于网络，但并行安装多个不冲突的应用
- 内存占用：不超过 500MB

### 5.2 安全要求
- 迁移包使用 AES-256-GCM 加密，密钥从迁移码派生
- 云端中转时服务器无法解密内容（端到端加密）
- 迁移码 24 小时后自动失效
- 密码和密钥类数据额外加密层
- 不收集、不存储任何用户数据

### 5.3 兼容性要求
- 支持 Windows 10（1903+）和 Windows 11
- 支持 x64 和 ARM64 架构
- MCP 协议兼容主流 Agent 平台

### 5.4 可靠性要求
- 文件传输校验哈希值，确保数据完整性
- 支持断点续传
- 单个步骤失败不影响其他步骤执行

## 6. 成功指标

| 指标 | 目标值 |
|------|--------|
| 应用识别率 | ≥ 95% |
| 自动安装成功率 | ≥ 80% |
| 配置恢复准确率 | ≥ 90% |
| 文件传输完整性 | 100% |
| 端到端迁移时间 | ≤ 2 小时（100GB 数据 + 50 个应用，千兆网） |
| 用户无需手动处理的比例 | ≥ 70% |
| 新电脑端设置时间 | ≤ 5 分钟（下载恢复器 + 输入迁移码） |

## 7. 术语表

| 术语 | 定义 |
